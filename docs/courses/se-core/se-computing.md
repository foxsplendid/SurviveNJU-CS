# 软件工程与计算

!!! info "以培养方案和当年课程通知为准"
    软件工程与计算（SEC）系列的课程编号、阶段划分、学分、开课学期、团队规模、项目周期、技术栈和评分指标可能随培养方案与教学班变化。本页不把往届项目经验写成当前硬性要求。请先查 [软件学院 2025 版本科培养方案入口](https://software.nju.edu.cn/rcpy/bks/20260514/i388724.html)，再以本人年级适用的正式方案和教务系统为准。

## 系列课程的能力主线

SEC 课程把程序设计训练与软件工程方法结合起来。不同阶段可能覆盖以下能力，但不应在缺少当年大纲时强行对应到 I、II、III：

- 面向对象程序设计、代码阅读、重构和单元测试；
- 需求获取、需求规格、建模与可追踪性；
- 架构与详细设计、接口契约和设计评审；
- 团队协作、版本控制、代码审查和持续集成；
- 测试、质量度量、交付、维护与演化。

是否采用 Web、移动端、微服务或其他技术栈，以及是否要求特定 UML 图、文档模板和部署平台，都应由当年任务书确认。

## 项目执行建议

1. **先拆评分规则和验收条件**：把任务书中的功能、质量、文档和过程要求转成可跟踪清单，不自行套用往届指标。
2. **建立可审查的协作流**：若课程要求团队开发，可采用短生命周期分支、Pull Request、代码审查和受保护主分支；具体流程服从课程与团队约定。
3. **保持需求、设计、代码和测试一致**：重要需求应能追踪到设计决策、实现提交和验证证据。
4. **持续集成而非期末拼接**：尽早运行构建、静态检查和测试，并记录环境和失败原因。
5. **解释指标而非追数字**：覆盖率、缺陷数和提交量需要结合风险与测试质量解释；不存在可跨年份套用的统一 `75%` 门槛。

!!! warning "项目材料边界"
    不要把往届团队的源码、报告或项目模板作为当前作业起点，也不要照搬其需求和设计文档。只使用课程组明确提供或许可的模板、依赖和示例。引用外部资料时保留来源；协作、复用与生成式 AI 的边界以当年书面规则为准。
