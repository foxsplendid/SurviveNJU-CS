# 操作系统：设计与实现 (OS)

*   **课程学分**：4 学分
*   **授课学期**：大二秋季学期
*   **授课教师**：蒋炎岩 (jyy) 等系统组教师
*   **修读阶段**：计算机系必修课

---

## 🌟 课程简介与地位

由蒋炎岩老师主讲的南京大学《操作系统：设计与实现》课程，在 CS 自学社区中评价很高，也常被 [csdiy.wiki](https://csdiy.wiki) 等自学指南推荐。它和 MIT 6.S081、UCB CS162 一样，适合作为操作系统方向的高强度实践课程参考。

不同于传统教材枯燥地背诵“操作系统的四大功能、死锁的四个必要条件”，这门课从全新的视角切入：

> **“什么是程序？程序就是状态机。”**
> **“什么是操作系统？操作系统就是一个由 API（系统调用）和中断驱动的软件对象。”**

你将在这门课中建立起真正的并发编程（Concurrency）思维，并深刻理解虚存、文件系统和进程调度的底层实现。

## 🛠️ 实验体系：OSLab & MiniLab

本课程的实验难度极大，包含两个并行开展的实验模块：

### 1. MiniLab (微型实验)
旨在让你通过少量的、极具技巧性的代码解决具体的操作系统核心概念问题：
*   **M1 (pstree)**：用 C 语言手写实现一个类似于 Linux `pstree` 的工具，通过读取 Linux `/proc` 文件系统打印系统进程树。
*   **M2 (libco)**：利用 `setjmp` / `longjmp` 手动在用户态实现一个轻量级、高效率的**协程库**。
*   **M3 (sperf)**：利用 `fork`、`execve` 和管道监控进程系统调用，手写一个系统调用 Profiler。
*   **M4 (crepl)**：手写一个交互式 C 语言 REPL 解释器，利用动态生成 C 源码编译并用 `dlopen` 运行时加载。
*   **M5 (frecov)**：解析磁盘映像 FAT 格式数据结构，手写工具恢复被删除的文件。

### 2. OSLab (操作系统大实验)
你需要在南大系统组开发的硬件抽象层 **Abstract-Machine (AM)** 上，从零实现一个支持对称多处理器（SMP）的真正操作系统内核：
*   **L0 (amgame)**：裸机小游戏。在没有 OS 支撑的环境下，直接在 Abstract-Machine 上开发一个图形小游戏，熟悉 bare-metal 裸机编程和 AM 调试。
*   **L1 (pmm)**：物理内存管理。在多处理器（SMP）裸机环境下，实现一个多线程安全的高效内存分配器（`kalloc` / `kfree`）。
*   **L2 (kmt)**：内核线程管理。基于 L1 内存分配，引入多处理器中断处理、时钟中断，并手写自旋锁（Spinlock）和信号量（Semaphore），支持抢占式多任务分时内核。
*   **L3 (uproc)**：用户态进程。基于 L2，实现虚拟内存页面映射，加载并隔离运行用户态程序，实现进程控制的系统调用（如 `fork`, `execve` 等）。
*   **L4 (vfs)**：（选做）虚拟文件系统。实现多线程安全的文件系统接口，支持文件的打开、读写与关闭。

## 💡 核心考点与学习秘籍

1.  **多线程并发编程**：
    *   **死锁（Deadlock）与数据竞争（Data Race）**是 OSLab 中最难调的 Bug。你必须在脑海中清晰画出各线程的状态转移图，严格遵守锁的获取顺序。
    *   利用多线程测试工具，如 `ThreadSanitizer` 来捕获潜在的竞争。
2.  **“程序是状态机”的思考模式**：
    *   无论是调试线程切换、中断响应，还是调试系统调用，都把计算机的行为看作是确定状态之间的跳转。一旦崩溃，立即追溯是哪一步跳转导致了状态机的异常。
3.  **不背概念，重在理解**：
    *   期末考试题目极其灵活，几乎没有“名词解释”和“简答题”。考题通常会给出一个精简的操作系统伪代码，让你分析其中是否存在死锁、并发漏洞或页表转换错误。
